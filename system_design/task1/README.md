# Система управления зоопарком

## Описание предметной области

Программа моделирует систему управления зоопарком:
- **Животные** - различные виды с уникальным поведением
- **Вольеры** - места содержания животных
- **Сотрудники** - работники зоопарка
- **Сервисы ухода** - кормление, уборка, лечение
- **Отчётность** - статистика и сводки по состоянию зоопарка

Система спроектирована с применением принципов SOLID.

---

## Применение принципов SOLID

### Single Responsibility Principle

Каждый класс выполняет одну задачу:
- `Zoo` — агрегирует компоненты зоопарка
- `Fence` — управляет вольером и его обитателями
- `FeedingService` — логика кормления
- `ReportService` — формирование отчётов и расписания

Это позволяет изменять логику отчетов, не затрагивая сервисы ухода.

### Open/Closed Principle

Система открыта для расширения, закрыта для модификации:
- Для добавления нового животного достаточно создать класс-наследник от `Animal`
- Для нового сервиса - реализовать соответствующий интерфейс
- Классы `Zoo`, `ReportService` и существующие сервисы не изменяются

### Liskov Substitution Principle

Все подклассы корректно заменяют базовые типы:
- Методы сервисов работают с абстракцией `Animal`, принимая любой объект (Lion, Snake, Parrot)
- Сотрудники корректно замещают абстрактный класс `Employee`
- Ни один подкласс не нарушает контракт базового класса

### Interface Segregation Principle

Вместо универсального интерфейса созданы узкоспециализированные:

**Для поведения животных:**
- `Speakable` - умеющие издавать звуки
- `Huntable` - хищники
- и другие по необходимости

**Для сервисов:**
- `ReportGenerator` - генерация отчётов
- Интерфейсы для кормления, лечения, уборки

Животные реализуют только нужные интерфейсы. Например, змея не реализует `Speakable`.

### Dependency Inversion Principle

Классы верхнего уровня зависят от абстракций:
- `ReportService` реализует интерфейс `ReportGenerator`
- `Zoo` работает с интерфейсами сервисов, не зная о конкретных реализациях
- Сервисы можно заменить на другие реализации без изменения `Zoo`

---

## Преимущества введённых абстракций

- Интерфейс `ReportGenerator` - позволяет заменить реализацию отчётов
- Разделённые интерфейсы поведения - животные не содержат пустых методов
- Абстрактный класс `Animal` - все сервисы работают с любыми животными через единый тип

---

## Возможные проблемы при масштабировании

- **Zoo хранит животных отдельно от вольеров** - животное может быть в списке `Zoo.animals`, но не быть ни в одном `Fence`. Нет синхронизации между этими коллекциями.

- **Zookeeper хранит ссылки на вольеры при создании** - если позже добавить новый вольер в зоопарк, смотритель о нём не узнает. Нет механизма обновления назначений.

- **Fence не проверяет совместимость животных** - можно поместить льва и попугая в один вольер. Нет валидации по `AnimalCategory` и нет ограничения вместимости по площади.
